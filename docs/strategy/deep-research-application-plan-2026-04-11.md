## 現状の整理

現状の `ExplainerCore` は、説明力、即答力、詰め対応、説得力という4能力の軸は適切に定義できている。一方で、今回の調査結果と照らすと、次の不足がある。

- 評価が主に「内容の構造」と「初動速度」に寄っており、会話ダイナミクス、長ポーズ、フィラー、自己認識差分の推移が弱い
- 即答力が 30 秒中心で、研究で重要だった 30 / 60 / 90 秒の時間設計と負荷段階がまだ薄い
- 詰め対応で「責任の取り方」「境界線」「不確実性の明示」は見ているが、圧の中での再着地設計と境界管理がまだ弱い
- 説得では相手視点を見ているが、利害関係者別の説明最適化、両面提示、接種理論型の反論先回りがまだ浅い
- 継続学習の設計が「回数」中心で、検索練習、間隔反復、ストレス下反復の設計がまだ十分ではない

## 推奨方針

### 1. 評価設計は「内容」「話し方」「対話」「自己認識」の4層に分ける

調査結果では、説明力や会話力の伸びを安定して捉えるには、単一総合点よりも観点分解の方が有効だった。したがって `Baseline Assessor` と各モジュールの評価は、最低でも次の4層で持つのがよい。

- 内容: 結論先行、構造化、要点充足、論点逸脱の少なさ
- 話し方: 長ポーズ、フィラー、話速、抑揚の単調さ
- 対話: 応答遅延、確認表現、相手理解、割り込み回避
- 自己認識: 自己評価と AI 評価の差分、その推移

### 2. モジュールは「短時間で回せる型」と「負荷段階」をセットで設計する

今回の研究では、即答や詰め対応は精神論ではなく、最初の一文と構造の自動化が最重要だった。各モジュールで「使う型」と「難度の上げ方」を固定するべきである。

- Baseline: `結論 -> 理由 -> 影響 -> 次アクション`
- Rapid Response: `結論 -> 現状 -> リスク/不確実性 -> 次アクション`
- Pressure Defense: `事実 -> 解釈 -> 不足情報 -> 是正策`
- Persuasion: `相手の利益 -> 提案 -> 懸念と対策 -> 合意したい次ステップ`

### 3. 学習運用は「反復回数」ではなく「反復の質」を上げる

調査結果では、想起練習、自己説明、If-Then、視点取得、低速呼吸、間隔反復の相性がよかった。したがって、各日の運用は「1つの warmup を選ぶ」だけでなく、「今日の弱点に応じた warmup を選び、その後に難度を1段だけ上げたシナリオを回す」流れに寄せるべきである。

## 優先度付きの反映提案

### P0: すぐ反映すべきこと

#### 1. ルーブリックを拡張する

最優先は `data/rubrics` の拡張である。理由は、採点観点が固定されると、シナリオや UI が後からでも揃えやすいからである。

- `baseline-assessment-rubric.json`
  - `metacognition` を追加し、自己認識差分を正式な観点にする
  - `interaction` を追加し、確認表現や応答の自然さを観る
  - `aggregateMetrics` に `longPauseCount` `fillerCount` `selfPerceptionGap` を追加候補として定義する
- `rapid-response-rubric.json`
  - 「30 / 60 / 90 秒の枠内で収める力」を観点として追加する
  - 「不確実性を短く明示して前に進める力」を追加する
- `pressure-defense-rubric.json`
  - 「責任の取り方」と「不用意な自己否定を避ける」を分けて見る
  - 「再着地できたか」を今より重く扱う
- `persuasion-rubric.json`
  - 「利害関係者別の説明調整」を明示的な観点にする
  - 「両面提示」と「反論先回り」を独立させる

#### 2. シナリオを難度別に増やす

次に `data/scenarios` を増やす。既存シナリオは方向性がよいが、パターン数がまだ少なく、反復で慣れやすい。

- `rapid-response-meeting.json`
  - 30 秒、60 秒、90 秒の3段階を追加する
  - `情報不足のまま答える` `経営層に短く答える` `オンライン会議で急に振られる` を追加する
- `pressure-defense-budget.json`
  - `責任追及` と `事実確認` と `感情的圧力` を分けたケースを追加する
  - `不当な詰め` に対して境界線を保つケースを追加する
- `persuasion-proposal.json`
  - `現場責任者向け` `上司向け` `他部署調整向け` の利害別シナリオを追加する
  - `メリットはあるがリスクが重い提案` を追加して両面提示を鍛える
- `baseline-interview.json`
  - `抽象的な説明` と `短い状況報告` と `優先順位変更` を分けて、説明の型の違いを測る

#### 3. warmup を能力別に固定する

`Evidence-Based Warmups` は、研究で裏づけが強かったものを中核に寄せるのがよい。

- わかりやすい説明: 自己説明プロンプト、抽象と具体の往復、聞き手別リライト
- 即答力: 30 秒想起練習、If-Then 即答プラン、30 / 60 / 90 秒切り替え
- 詰め対応: 低速呼吸、事実と解釈の分離練習、境界線つきの再着地練習
- 説得力: 視点取得、両面提示、反論先回りの一言目練習

### P1: 次に反映すべきこと

#### 4. セッション記録に「会話ダイナミクス」を入れる

将来の `coach-web` では、内容採点だけでなく、以下の計測値を持てると研究結果との整合性が高い。

- `responseLatencyMs`
- `longPauseCount`
- `fillerCount`
- `selfPerceptionGap`
- `followUpHandlingScore`

#### 5. Dashboard は「過去の自分比較」を前面に出す

調査結果では、継続利用には他人比較より自己比較が有効だった。したがって `Session Review Dashboard` では次を優先するとよい。

- 直近3回平均との差分
- 自己評価と AI 評価の差分推移
- 能力別の改善率
- よく崩れるポイントの自動要約

#### 6. 週間運用を「間隔反復」に寄せる

現在の週間運用は妥当だが、同じ能力を連日固定で回すより、1日置きに再測定する設計の方が研究結果と合う。

- 月曜: Baseline or 説明
- 火曜: 即答
- 水曜: 説得
- 木曜: 詰め対応
- 金曜: 即答か説明の再測定

### P2: 補助テーマとして扱うこと

#### 7. ユーモア会話は独立モジュール化せず補助訓練に留める

ユーモア研究は有益だが、現時点のプロダクト主目的は「社内説明力」と「会議会話力」である。したがって、いまは独立モジュール化よりも次の形がよい。

- warmup の補助テーマとして扱う
- 雑談やラポール形成の一言目テンプレートとして少量反映する
- 本流の rubric には混ぜすぎない

## いま着手するなら何をやるべきか

1. `data/rubrics` を先に更新する
2. `data/scenarios` を 30 / 60 / 90 秒と利害別に増やす
3. `docs/evaluation` と `docs/training` を新ルーブリックに合わせて更新する
4. その後に `coach-web` 側のメトリクスと表示を広げる

3. 評価文書と訓練文書の更新

## 推奨結論

今回の調査結果を踏まえると、最も効果が高いのは「まず評価軸を研究準拠に寄せること」である。新機能追加より先に、何を良い応答と定義するかを明確にし、その後にシナリオ、運用、UI を揃える進め方を推奨する。
