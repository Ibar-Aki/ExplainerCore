# Project Charter — ExplainerCore の目的とスコープ

> **迷ったときに戻る場所です。** 機能追加や方針変更を検討するとき、まずここに照らし合わせてください。

- 最終更新: 2026-04-19

---

## このプロジェクトが存在する理由

仕事での「説明力」と「会話力」は、感覚に頼るだけでは伸びにくいスキルです。  
ExplainerCore は、**繰り返せる訓練の仕組みそのものを整備する**ために作られました。

特に以下の3つの場面に絞って鍛えます。

1. 会議で急に話を振られたときの **即答力**
2. 厳しい追及を受けたときの **防御力（論点維持力）**
3. 相手を動かすための **説得力**

---

## 対象者

- 日本語で社内会議や報告を行うビジネスパーソン
- 会議での説明に苦手意識はあるが、訓練で改善したいと考えている人
- その場で考えを整理して伝える必要がある人

---

## 成功条件

以下がすべて満たされていれば、このプロジェクトは機能しています。

- 4つの能力（説明力・即答力・詰め対応・説得力）それぞれに明確な評価軸がある
- 週単位で訓練を回せる
- 訓練結果が履歴として残り、以前と比較できる
- 自己評価とAI評価の差分が見える
- 使い続けても記録が散逸しない

---

## 範囲

### 含むもの
- 調査プロンプト、採点ルーブリック、訓練シナリオ
- ローカルWebアプリ（`apps/coach-web`）
- セッション記録と振り返りテンプレート

### 含まないもの
- 企業向けのマルチユーザー運用
- クラウド同期
- 英会話・一般スピーチの訓練
- 外部会議システムや CRM との連携

---

## 重要原則

| 原則 | 理由 |
|---|---|
| 日本語の社内会議・報告に絞る | スコープを広げすぎると評価軸がぶれる |
| 音声とテキストの両方を扱う | 実戦に近い記録ができる |
| AI採点に依存しすぎない | 自己レビューと履歴比較も必ず残す |
| 学習素材より実戦再現を優先する | 知識より行動の自動化が目的 |
| Provider設定状態と実行結果を混同しない | Remote失敗時の誤解を防ぐ |
| UI/UX変更は必ず文書に反映する | 設計意図の断絶を防ぐ |

---

## 判断基準

機能追加や変更を検討するとき、以下の順で判断してください。

1. 実戦の訓練に直接効くか
2. 継続運用時に記録と比較がしやすいか
3. 実装とドキュメントの整合が保てるか
4. ローカル環境で無理なく回るか
